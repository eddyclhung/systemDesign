# Payment System

**Hard** · Vol 2 · Ch.11 · Idempotency · Double-entry ledger · Reconciliation

Tags: `Idempotency`, `Stripe`, `Double-entry`, `Reconciliation`, `Saga`, `Distributed Transactions`

**Flow:** User → Payment service → PSP (Stripe/Adyen) → Card network → Bank → Settlement → Ledger update → Notification

---

<details open>
<summary><strong>Problem</strong></summary>

Design a payment processing system that handles money movement correctly even under network failures, retries, and distributed system failures. The core challenge: money must not be double-charged, never lost, and every cent must be accounted for.

</details>


<details>
<summary><strong>Key points</strong></summary>

- **Idempotency key** — Every payment request carries a unique idempotency_key (UUID). On retry: server returns same result as original request. Prevents double charge. PSP (Stripe) also honors idempotency keys end-to-end.
- **Payment state machine** — Pending → Processing → Succeeded / Failed / Cancelled. Transitions are atomic DB updates. Each state is observable for debugging. Never skip states.
- **Double-entry ledger** — Every payment = debit one account + credit another. Sum of all ledger entries always = 0. Immutable — entries are never updated, only new entries created. Complete audit trail.
- **PSP integration** — Never process card data yourself — PCI DSS compliance is extremely complex. Use a PSP (Stripe, Adyen, Braintree). They handle card networks, encryption, compliance. You send payment intent, they handle the rest.
- **Idempotency at PSP level** — PSPs support their own idempotency keys. Even if your system retries, the PSP deduplicates on their end. Two-layer protection: your idempotency_key check + PSP's idempotency_key.
- **Reconciliation** — Daily batch job compares your ledger against PSP settlement statement. Flag discrepancies. Human review for unmatched records. This is the source of truth check.
- **Asynchronous webhooks** — PSPs notify you of payment outcomes via webhooks. Don't rely on synchronous API response alone — webhook may carry different final status than the initial response.

> Idempotency key prevents double charges. Double-entry ledger ensures books balance. Daily reconciliation catches any discrepancies. These three together make a correct payment system.

</details>


<details>
<summary><strong>Scale</strong></summary>

Payment processing is not a throughput problem — it's a correctness problem. 115 transactions/sec is well within a single PostgreSQL instance's capacity.

The scaling concern is consistency under failure. The canonical failure mode: payment request times out. Did the charge succeed? The idempotency key answers this definitively. This is why idempotency is the first concept to introduce — it makes the system correct under retries, which is what makes the system correct under real network conditions.

</details>


<details>
<summary><strong>Script</strong></summary>

1. Financial integrity framing.
2. "Payment systems must satisfy two invariants: no double charges, and no missing money. The design flows from these constraints."
3. "No double charges: every payment request carries an idempotency_key (UUID generated client-side). On retry: server returns same result. PSP honors our idempotency_key. Two-layer protection."
4. "No missing money: double-entry ledger. Every transaction = debit one account + credit another. Sum of all entries = 0. Immutable — no updates, only new entries. Complete audit trail."
5. "Payment flow: Payment service → PSP (Stripe) → card network → bank. PSP webhooks notify us of final status. Poll PSP API as fallback (don't rely solely on webhooks)."
6. "Daily reconciliation: compare internal ledger against PSP settlement statement. Any discrepancy → alert for human review. This is the source of truth check."

</details>
